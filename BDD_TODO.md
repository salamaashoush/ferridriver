# BDD Framework TODO

## Session Summary

Built a complete BDD/Cucumber/Gherkin framework for ferridriver across 3 new crates, extended the core test engine with step-level events, added CLI support in both Rust and TypeScript, and verified 32 scenarios passing identically on both CLIs at ~5.5s with 6 workers.

### What Was Built

**3 new crates:**
- `crates/ferridriver-bdd` -- BDD/Cucumber/Gherkin library (parsing, 109 built-in steps, cucumber expressions, step registry, translation layer, reporters, world)
- `crates/ferridriver-bdd-macros` -- proc macros (`#[given]`, `#[when]`, `#[then]`, `#[step]`, `#[before]`, `#[after]`)
- `examples/bdd-example` -- working example with custom steps + `bdd_main!()`

**Core engine extensions (ferridriver-test):**
- `StepStarted`/`StepFinished` real-time reporter events
- `TestInfo::begin_step()` -> `StepHandle` -> `handle.end()` API
- `TestStep` extended with `step_id`, `status: StepStatus`, `location`, `parent_step_id`, `metadata: Option<serde_json::Value>`
- `HookFn` now receives `(FixturePool, Arc<TestInfo>)` -- hooks see tags, test name, step API
- `SuiteHookFn` for `before_all`/`after_all` (no test context)
- `context` injected into fixture pool alongside browser/page/test_info
- All 4 reporters (terminal, JSON, JUnit, HTML) handle step events with hierarchy
- Page expect: `to_have_screenshot`, `to_match_aria_snapshot`, `to_contain_title`, `to_contain_url`
- Locator expect: `to_have_accessible_error_message`
- Screenshot pixel diff extracted to shared `snapshot::compare_screenshot_png()`

**Core API fixes (ferridriver):**
- `CookieData` -- added `same_site: Option<SameSite>` (Strict/Lax/None) to all 3 backends
- `SetCookieParams`, `ClearCookieOptions` types added
- `Page::with_context()` + `page.context()` returning `Option<&ContextRef>`
- `ContextRef::clear_cookies_filtered()`, `ContextRef::delete_cookie()`
- `PLAYWRIGHT_COMPAT.md` tracking remaining API gaps

**CLI:**
- Rust: `ferridriver bdd [OPTIONS] -- [FEATURES...]` with --tags, --dry-run, --list, --workers, --headed, --grep, etc.
- TS: `ferridriver-test --bdd --features ... --steps ... --tags ...` with step file auto-discovery

**NAPI:**
- `BddRunner` class with `registerStep(kind, pattern, callback)`, `registerHook(point, scope, callback)`, `run()`
- Built-in Rust steps (109) linked via inventory into NAPI binary
- TS API: `Given()`, `When()`, `Then()`, `Step()`, `Before()`, `After()`, `BeforeAll()`, `AfterAll()`, `configureBdd()`, `runFeatures()` in `@ferridriver/test/bdd`

**Architecture (key design decisions):**
- BDD crate is a thin translation layer -- translates Features->TestPlan, core `TestRunner` does all execution (no duplicate runner/dispatcher/worker)
- Step events are generic -- `StepCategory::TestStep` for all steps, BDD keyword info in `metadata` (generic `serde_json::Value`), not domain-specific enum variants
- Assertions use ferridriver-test's expect API with auto-retry (`poll_until`) -- no manual polling or hardcoded sleeps
- `bdd_main!()` macro uses core `TestRunner` directly -- same worker pool, parallel dispatch, retries, reporters as E2E/CT tests
- ONE config system (`TestConfig`), ONE runner, ONE set of reporters

**Test results:** 32 scenarios across 8 feature files, 31 passed, 1 skipped (@skip tag), identical on both Rust and TS CLIs at ~5.5s with 6 workers
